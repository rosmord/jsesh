# ADR 2: introduce a CanonicalCode interface
 
Status
: accepted

## Context

The classes implementing `HieroglyphShapeRepository` had a `ShapeChar get(String code)` method which didn't specify what a code was. Each `HieroglyphShapeRepository` had to normalise the codes before looking for the corresponding shape. This was error prone. 
`

## Decision

Enforce normalisation through the introduction of a `CanonicalCode` interface. The only way to create a `CanonicalCode` is through the `ManuelDeCodage` class, which knows how to normalise codes.

## Consequences

- normalisation is **explicitly** enforced by the type system;
- the normalisation is now performed by the **clients** before calling the `get` method, which is a bit annoying;
    - we could improve things by keeping both the normalised code and the original in the representation of glyphs - for speedup;
    - we could also introduce a facade which would deal with normalisation, but it would be an extra layer.
