# ADR 3 : rationalise the creation of resources

Status
: proposed

## Context

To render or edit MdC text, callers need three kinds of resource along two source axes:

- **shapes** (`HieroglyphShapeRepository`) — drawable glyphs, from embedded fonts + a user folder;
- **database** (`HieroglyphDatabase`) — sign codes, values, families, variants, from embedded XML + a user file;
- **possibilities** (`PossibilityRepository`) — transliteration → sign or sign-sequence, from a user file or from the database.

They are bundled into an aggregate, `HieroglyphToolkit`, and handed to `JMDCEditor` / `JMDCField`.
The last refactoring removed the JSesh v7 singletons, but left the *creation* side confusing — even to us:

- **six** overlapping ways to obtain the aggregate, with no obvious entry point and no signal of shared-vs-fresh (`standardHieroglyphToolKit()`, `embeddedOnlyInstance()`, two `buildFontWith…` factories, a public constructor, `new JSeshUserSignLibraryConfiguration()`);
- `HieroglyphToolkit` is an interface with two implementations that add **no polymorphism** — they only differ in how they are assembled; and `JSeshUserSignLibraryConfiguration` leaks `glossaryManager()`, forcing downcasts;
- inconsistent / misleading names (`standardHieroglyphToolKit`, `buildFontWithoutUserDefinitions` builds a whole toolkit not a font, `PredefinedFonts.compositeFont()`, `get`-vs-no-`get` accessors);
- I/O and eager singletons in constructors (`GlossaryManager`, `JSeshFullHieroglyphShapeRepository`, `StandardFontShapeRepository`), hurting testability;
- the subsystem is scattered across five packages.

Full analysis: [`../audit/20260709-resources_refactoring.md`](../audit/20260709-resources_refactoring.md).
- `PossibilityRepository` is built from a `HieroglyphDatabase` **and** a `Glossary`. it's a bit annoying, because either:
  - we had the glossary, which has no business here, in the fundamental resources ;
  - or we get problems building the resources, because we need the glossary (and the database) to build the possibilities.
- the most logical approach is to **remove** the `PossibilityRepository` from the fundamental resources, and manage it separately.
- as often, the main problem is to understand what we want to do when using the **default constructor** of `JMDCEditor` / `JMDCField`. What `PossibilityRepository` should be used in this case? If we want a shared one, we might need to have a shared instance, which itself would be connected to the shared `HieroglyphResources`.

## Decision

1. **Resources are resources.** Replace the `HieroglyphToolkit` interface *and* both implementations
   (`SimpleHieroglyphToolkit`, `JSeshUserSignLibraryConfiguration`) with a single immutable record
   **`HieroglyphResources`** holding `shapes`, `database`, `possibilities`.
   The record is enough because the real variability lives one level down, in the
   `HieroglyphShapeRepository` / `HieroglyphDatabase` interfaces, which stay pluggable.

2. **One construction entry point:** a `HieroglyphResourcesLoader` (fluent builder along the two
   axes: `addFont`, `userFontDirectory`, `userSignDefinitions`, `userGlossary`, `build()`), plus two
   named presets — `embedded()` and `fromUserPreferences()` — replacing the six current paths.
   All disk I/O happens in the loader, so the resource objects gain plain data-in constructors and
   become testable.

3. **Naming convention** for shared vs. fresh: `getXxx()` / `xxxInstance()` = a process-wide
   singleton (reserved for the immutable embedded font resources); `build()` / `embedded()` /
   `fromUserPreferences()` / `new` = a fresh object. Fix the stray-`K` `standardHieroglyphToolKit`,
   drop "Font" from the factory names, and remove `PredefinedFonts.compositeFont()`.

4. **Keep `JSeshRenderContext` separate** (it is the *drawing* concern = style + shapes; ~30
   drawing/export files need it without database or glossary). Join the two with an additive bridge:
   `HieroglyphResources.renderContext(JSeshStyle)`.

5. Split `GlossaryManager` (holder + reader + writer + paths) into an in-memory `Glossary` and an
   explicit storage gateway with `load()` / `save()`; give the subsystem a home package
   (e.g. `jsesh.resources`) and move `PossibilityRepository` there from `jsesh.editor`.

## Consequences

- One obvious, well-named way to get resources; the two old implementations and six factories collapse
  to one record + one loader.
- Flexibility is **preserved**: custom fonts, custom databases, and future "sign libraries"
  (Old Kingdom vs. Ptolemaic) are just recipes on the loader.
- Testability improves: with I/O in the loader, `HieroglyphResources` can be built from temp files with
  no dependency on the real user preferences.
- Churn is concentrated and mostly mechanical: `HieroglyphToolkit` consumers (~22 files, mostly
  `jseshSearch`) change type name; `JSeshUserSignLibraryConfiguration` touches only the two app
  bootstraps and their wiring; `JSeshRenderContext` / `JSeshStyle` are untouched (bridge is additive).
- Open point: whether `HieroglyphResources.glossary` is a `GlossaryManager` or `Optional<GlossaryManager>`
  (for embedded kits with no glossary).
